# Korean Prose

Use this reference for Korean drafting, Korean naturalization, grammar and
spacing cleanup, Korean AI-pattern review, and Korean publication prose.

## Core Rule

한국어 문장은 의미와 출처를 보존하면서 문서 성격에 맞는 격식, 용어, 맞춤법,
문법, 띄어쓰기, 문장부호를 갖춰야 한다. 번역투와 AI 문체는 검토 신호로
사용하되, 기술 식별자나 의도적인 표현을 과교정하지 않는다.

한국어를 자연스럽게 고치는 일은 내용 수정이 아니다. 주장, 범위, 조건, 책임
주체, 인과관계, 부정, 기술 용어가 바뀌면 다시 쓰기가 실패한 것이다.

## Meaning Audit

다시 쓴 뒤에는 다음 항목을 확인한다:

- 이름, 제품명, 기관명, 프로젝트 키;
- 숫자, 비율, 금액, 버전, 기간, 단위;
- 날짜, 시간, 시간대, 순서;
- 직접 인용과 source-bound 표현;
- 법률, 규정, 조항 문구;
- 코드, API, CLI, 파일 경로, 이슈 번호, 모델명;
- 가능성, 추정, 단정, 의무, 권고의 층위;
- 원인과 결과, 조건과 결과;
- 주체와 행위자;
- `일부`, `대부분`, `모든`, `필수`, `선택` 같은 범위 표현;
- 긍정, 부정, 예외 조건;
- 원문 정보의 누락이나 원문에 없는 설명의 추가.

하나라도 바뀌면 더 자연스러운 문장이라도 원문 의미를 복원한다.

근거 없는 의미 부여는 자연스러운 한국어로 다듬을 대상이 아니다. 원문이나
제공된 근거가 뒷받침하지 않는 `중요한 전환점`, `상당한 가치`, `확장성 확보`,
`협업 강화`, `체계적인 발전`, `큰 효과` 같은 표현은 삭제하거나, 근거가
필요하다는 finding으로 남긴다. 더 약한 표현으로 바꾸더라도 근거 없는 효과나
가치를 새로 만들면 실패다.

## Register And Terminology

문서의 격식을 먼저 정한다:

- 기술 문서는 명확한 설명체와 절차형을 우선한다.
- 비즈니스 공유 문서는 정중한 격식체를 쓸 수 있다.
- 블로그나 외부 발행물은 매체 voice를 따를 수 있다.
- 회의 메모, 캡처, 인용문은 자연스러운 register variation을 보존할 수 있다.

한 문서 안에서 `합니다`, `한다`, `해요`, 반말이 무작위로 섞이면 독자가 문서
성격을 오해한다. 다만 자연스러움을 위해 문서 전체 격식을 바꾸지 않는다.

같은 개념은 같은 용어로 쓴다. `사용자`, `유저`, `고객`을 바꾸어 쓰면 서로
다른 개념처럼 보일 수 있다. 공식 제품명, API, SDK, CLI, repository 이름,
파일 경로, 속성명, 명령어는 한국어 맞춤법보다 owning source 표기가 우선한다.

## Grammar, Spacing, And Punctuation

우선 고칠 항목:

- `되/돼`, `안/않`, `던/든`, `웬/왠`, `-ㄴ지/-는지`, `-ㄹ게/-를게`;
- 조사와 명사의 호응;
- 어미, 시제, 높임 표현의 문맥상 자연스러움;
- 의존명사와 보조용언 띄어쓰기;
- 단위, 수량, 날짜, 시간 표기의 문서 내 일관성;
- 쉼표, 콜론, 따옴표, 백틱의 용도 혼동.

확신이 낮은 표기 문제는 필수 finding으로 단정하지 말고 확인이 필요한
제안으로 남긴다. 기술 식별자와 프로젝트 관례는 일반 표기 정규화보다 우선할
수 있다.

## Translationese And Korean AI Patterns

다음 표현은 금지어가 아니라 검토 신호다. 문장이 장황하거나 부정확하거나
근거 없이 격식 있어 보일 때 국소적으로 고친다.

| Signal                                     | Risk                      | Prefer                           |
| ------------------------------------------ | ------------------------- | -------------------------------- |
| `~에 대해`, `~에 있어서`, `~와 관련하여`   | 영어식 전치사 구조        | 직접 목적어나 서술어             |
| `~를 통해`, `~에 기반하여`, `~을 바탕으로` | 수단, 근거, 조건 혼동     | 실제 관계를 명시                 |
| `가지고 있다`, `되어진다`, `~에 의해`      | 불필요한 동사와 이중 피동 | 단순 동사나 단순 피동            |
| `~할 수 있다` 반복                         | 조건과 실제 동작이 흐려짐 | 조건 또는 확정 동작              |
| `~할 것이다`, `기대된다` 반복              | 근거 없는 미래 단정       | 확인된 계획, 예상, 가설 구분     |
| 추상 한자어 과다                           | 구체성이 낮아짐           | 구체적 대상과 동사               |
| 3박자 구조 반복                            | 생성문 리듬               | 필요한 항목만 유지               |
| `-들` 과다                                 | 영어 복수형 직역          | 복수가 필요할 때만 사용          |
| `그`, `이`, `해당`, `이러한` 반복          | 주체가 흐려짐             | 명사구 반복 또는 자연스러운 생략 |
| 접속사 과다                                | 모든 문장이 같은 흐름     | 필요한 논리 전환만 유지          |

한국어에서는 자연스러운 주어 생략이 가능하다. 모든 문장에 주어를 넣어 영어
문장 구조를 따라가지 않는다.

## Naturalization Discipline

AI 티를 줄이거나 번역투를 줄일 때:

1. protected span을 먼저 표시한다.
2. 문제가 있는 문장이나 구만 고친다.
3. 문서 전체 구조와 주장 순서는 사용자가 요청하지 않으면 유지한다.
4. 격식은 유지하되 문장 길이와 종결 리듬만 필요한 만큼 조정한다.
5. 큰 rewrite 뒤에는 의미 감사를 다시 한다.
6. 잔여 문제가 구체적으로 없으면 두 번째 rewrite를 하지 않는다.

실질 표현 변경이 크면 이유를 설명하고 범위를 줄인다. 문서 전체 tone, 구조,
주장 순서를 바꾸는 작업은 사용자가 요청했거나 owner workflow가 허용한 경우에만
한다.

## Publication-Specific Korean

외부 발행물이나 블로그 초안에서는 독자, 매체 voice, citation style, narrative
flow가 더 중요할 수 있다. 그래도 통계, 공식 사양, 릴리스 날짜, 외부 주장에는
출처가 필요하고, 빈 강조어와 추상 hype는 근거 없이 쓰지 않는다.

Publication 규칙을 canonical 문서나 owner workflow draft에 자동 적용하지
않는다.

## False Positives

다음은 자동 결함이 아니다:

- 인용문 안의 어색한 표현;
- 기술 식별자나 공식 용어;
- 회의 메모와 rough capture에서 나타나는 자연스러운 문체의 흔들림;
- 도메인상 필요한 영어 용어;
- 이미 명확한 짧은 문장;
- publication voice로 의도된 리듬.

확신이 낮으면 rewrite보다 finding으로 남긴다.
